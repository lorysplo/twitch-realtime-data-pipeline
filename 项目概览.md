# 项目概览：Twitch 实时弹幕数据管道

## 1. 项目概述

这是一条端到端的数据管道，数据源是 Twitch 直播间的实时聊天弹幕，从采集一直做到 BI 报表，跑在 AWS 上，实时和离线两条处理路径并行。

做这个项目的目的，是用一个真实、公开、高吞吐、又能模拟实时流的数据源，把数据工程的整套链路——流处理、数据湖、建模、数仓、BI、编排——完整打通一遍。选 Twitch 弹幕，是因为它量大（头部主播每秒几十条）、有天然的流量尖峰（主播被 raid 突袭、电竞高光时刻），字段也丰富（弹幕之外还有 emote、打赏、订阅、突袭事件），而且每条带高精度时间戳，可以按原始节奏确定性回放。

## 2. 整体架构

走 Lambda 架构，一个数据源，分实时和离线两条路：

```
两个采集源 → 回放推进 Kinesis(两条流)
   ├ 实时(速度层)：Lambda 算窗口指标+情感 → DynamoDB → 大屏          看「当下」
   └ 离线(批量层)：Firehose→S3 → Spark 打分 → dbt 建模 → Redshift → QuickSight   看「全量/趋势」
                                  ↑ 由 Airflow 编排调度
```

- 实时这条延迟低、结果近似，回答「现在这一秒什么状况」。
- 离线这条延迟高、结果精确、可重算，回答「这场/这几天的准确总账」。
- 两条共享同一个数据源；离线是真相源，实时是快速近似。

用到的组件，逐个说明（多数是 AWS 的服务）：

- **Kinesis**：AWS 的实时数据流服务，数据像水流一样实时经过，这里用来接弹幕和在线人数。
- **Lambda**：AWS 的无服务器函数，有数据来就自动跑一段代码，这里用来算实时指标。
- **DynamoDB**：AWS 的键值数据库，按 key 秒级存取，这里存实时算好的指标给大屏读。
- **Kinesis Firehose**：AWS 的服务，把流里的数据一批批打包成文件、存进 S3。（注意：它是个搬运工具，不是直播间。）
- **S3**：AWS 的文件存储，相当于云上的硬盘，这里当数据湖，原始和加工后的数据都堆这。
- **Spark / Glue**：Spark 是分布式计算引擎，Glue 是 AWS 托管的 Spark；这里用来给海量弹幕跑情感打分。
- **dbt**：用 SQL 做数据清洗和建模的工具，这里把数据建成星型模型。
- **Redshift**：AWS 的数据仓库，存建好的模型，供 BI 快速查询。
- **Athena**：直接对着 S3 上的文件跑 SQL 的服务，不用先把数据导进数据库。
- **QuickSight**：AWS 的 BI 工具，把数据画成图表和看板。
- **Airflow**：编排调度工具，按时间和依赖自动把离线那一串任务跑起来。

## 3. 数据源与字段

两个独立的采集源，从一开始就分开（节奏和结构都不同）：

**源一 · Twitch IRC（弹幕主源）**，每秒几十条。解析两类消息：

| 字段 | 含义 |
|------|------|
| ts | 高精度时间戳（回放靠它） |
| channel / room_id | 频道名 / 房间 ID |
| msg_type | chat / cheer / sub / resub / subgift / raid … |
| user_id / display_name | 用户 ID / 显示名（脱敏前） |
| text / emotes | 弹幕内容 / emote |
| is_sub / is_mod | 是否订阅者 / 管理员 |
| bits | 打赏 bits 数 |
| event_info | 事件细节（订阅月数、突袭人数等，仅事件有） |

**源二 · Twitch Helix API（在线人数）**，每分钟轮询一次：

| 字段 | 含义 |
|------|------|
| ts / channel | 时间戳 / 频道 |
| viewer_count | 当前观看人数 |
| game / title | 当前游戏 / 标题 |

数据量：约 30 万条弹幕 + 在线人数心跳，跨 4 天，覆盖 8 个头部频道。

## 4. 数据流向

1. **采集**：匿名连 Twitch IRC 抓弹幕/事件，写带时间戳的 jsonl；同时用 Helix API 每分钟轮询在线人数，写另一份 jsonl。
2. **回放成流**：按相邻两条记录的时间差逐条吐出，推进 Kinesis——弹幕进 `twitch-chat-stream`，人数进 `viewer-counts`，partition key 用 `room_id` 保证同频道有序。
3. **实时路**：Kinesis 触发 Lambda，算窗口指标写 DynamoDB，大屏读 DynamoDB 展示。
4. **离线路**：Firehose 把流落进 S3，Spark 打情感分，dbt 建模落 Redshift，QuickSight 出报表；整条由 Airflow 每天调度。

## 5. 实时链路

实时这条用的是 Amazon Kinesis 接数据流、Lambda 做计算、DynamoDB 存结果、前端大屏读 DynamoDB。

我们往实时里送两类数据：一类是聊天弹幕，连带打赏、订阅、突袭这些事件；另一类是直播间的在线人数。这两类分成两条 Kinesis 流来走——弹幕一条、人数一条。

之所以分开，是因为这两类数据根本不是一回事：弹幕每秒几十条、结构都一样，靠一个 `msg_type` 字段就能区分是普通弹幕还是打赏、订阅；在线人数每分钟才来一条、字段也完全不同。要是硬塞进同一条流，下游每读一条都得先判断「这是弹幕还是人数」，还白占一份流的成本。所以要不要拆流，看的是数据结构、来的频率、谁来消费——三样不一样就拆。（顺带说一句，不会按「直播间」去拆流，频道之间用 `room_id` 分区键区分就行；真按频道拆，频道一多流就管不过来了。）

数据进了 Kinesis，弹幕流触发 Lambda，按滚动时间窗口算几个指标——每秒消息数、突增告警、当前情感、最近一段最火的 emote、打赏速率；人数流给出实时在线曲线。算完写进 DynamoDB，大屏轮询 DynamoDB 显示。

要说明的是，实时这条只看最近一个窗口、秒级出结果、而且是近似值，算完就丢、只留聚合。它不做数据分层、也不做维度建模——大屏要的只是「按一个 key 立刻取到最新算好的数」，不是多维度的分析查询。分层和建模都在下面的离线链路里。

## 6. 离线链路

离线这条用的工具是：Kinesis Firehose 把流落成文件、S3 当数据湖、Spark（跑在 Glue 上）做重计算、dbt 建模、Redshift 当数仓、最后 QuickSight 出报表。

先说 Firehose 是干嘛的。实时那两条 Kinesis 流是「流过就没」的，离线要的是把它们攒下来。Firehose 就是接在同一条流后面的搬运工，每隔几分钟、或者攒够一定量，就把这段数据打包成文件写进 S3。两条流（弹幕、人数）各接一个 Firehose，分别落到 S3 的不同目录。

这里顺便澄清一下「直播间」的事：我们抓了 8 个头部频道，但它们不是 8 套独立的管道——所有频道的弹幕都混在同一条弹幕流里，靠记录里的 `channel` / `room_id` 字段区分，Firehose 一起落盘。频道多，只是数据行数多，不影响这条链路的结构，也不用每个频道单独建东西。

数据落到 S3 之后：Spark 把每条弹幕读出来打情感分（这步是分布式 Python 计算，SQL 干不了）；dbt 把打过分的数据清洗、脱敏、建成星型模型，落进 Redshift；QuickSight 连 Redshift 出报表——频道日报、营收、情感趋势、人数曲线、激增分析。

下面的「数据分层」和「数据建模」都只发生在这条离线链路里。

### 6.1 数据分层（Bronze / Silver / Gold）

- **Bronze**：原始 jsonl 一字不动地落 S3。留底，任何指标都能从这里重算。
- **Silver**：清洗、去重、`user_id` 脱敏、加情感分、定类型、按日期分区，存 Parquet。
- **Gold**：维度建模后的星型 fact/dim 表，面向 BI 消费，落 Redshift。

### 6.2 数据建模（Gold 层）

Gold 层是星型模型，三个事实表共享一组维度：

```
        dim_date    dim_user
            \        /
 dim_channel ── fact_message       一条弹幕
 dim_channel ── fact_event         一个事件(订阅/突袭)
 dim_channel ── fact_viewer ── dim_game   一条人数心跳
```

- 事实表（按粒度）：`fact_message`（一条弹幕，度量 bits/情感/字数）、`fact_event`（一个事件，度量突袭人数等）、`fact_viewer`（频道×分钟的人数快照）。
- 维度表：`dim_channel`、`dim_user`、`dim_date`、`dim_game`。
- 订阅状态（is_sub）会随时间变，属于缓慢变化维（SCD）。这里简化处理：`dim_user` 记「是否曾订阅」，每条弹幕当时的订阅状态作为退化维度记在 `fact_message` 上。

**为什么这么建模**

模型是从「我们最终要回答什么业务问题」和「数据本身长什么样」这两头推出来的，不是套模板。

我们要出的分析大致是这几类：每个频道每天的健康度（消息量、活跃用户数、人均发言、订阅占比）、频道营收（打赏 + 订阅）、观众情绪走势、在线人数和弹幕活跃度的关系、raid 激增的影响、用户分层。这些分析有个共同点——基本都是「**按某个频道、某一天（有时再按用户或游戏）去看一组指标**」。

再看数据源的情况：一是分析的主体天然就是「频道」和「用户」，时间（天/小时）是贯穿所有分析的轴；二是数据本身是三种不同形态——每秒几十条的弹幕、零散发生的生命周期事件（订阅/续订/突袭）、每分钟一次的在线人数快照，三者节奏不同、能算的度量也不同。

于是模型就这么定下来：

- 既然几乎所有分析都按 **频道 / 用户 / 时间 / 游戏** 切，这四个就自然成为维度（`dim_channel` / `dim_user` / `dim_date` / `dim_game`），并做成三张事实表共享的维度，保证「按频道」「按天」在所有报表里口径一致、能交叉对比（比如弹幕量和在线人数按同一频道同一天 join 起来看相关性）。

- 既然数据是三种不同形态、不同粒度，就拆成三张事实表各对一种：`fact_message`（弹幕，算消息量/情感/参与度）、`fact_event`（订阅/突袭，算营收/激增）、`fact_viewer`（人数快照，算在线趋势）。强行塞进一张表，粒度一混聚合就会错（把人数和弹幕条数加一起没有意义）。其中 `fact_viewer` 是「每分钟量一次」的周期快照，它的 viewer_count 不能跨时间求和、只能取均值/最值。

- 弹幕里带的字段正好对应这些业务度量，所以直接放进 `fact_message`：bits → 营收，sentiment → 情绪，is_sub → 群体行为，emote/字数 → 参与度。

- 用户要做分层/留存，所以建 `dim_user`；但 user_id 是 PII，入维度前先脱敏成不可反查的 key。订阅状态会变（SCD），不过目前的分析不需要追「某用户哪天从非订阅变订阅」，就简化成存「是否曾订阅」+ 每条弹幕记当时状态，够用；真要做订阅生命周期分析再上 SCD2。

## 7. 编排与调度（Airflow）

离线是一条按天跑的链，步骤之间有依赖（dbt 要在 Spark 打分、Glue 建表之后才能跑），且任一步都可能失败。Airflow 把它定义成一张 DAG：按依赖决定顺序，失败自动重试、告警，并能对某天单独补跑。cron 不管依赖、不管失败，做不到这些。

DAG 每天凌晨跑一次：

```
upload_bronze → spark_sentiment → crawl_bronze → build_warehouse → build_snapshot → refresh_dashboard
```

- **upload_bronze**：当天原始弹幕落进 S3。
- **spark_sentiment**：Glue 上的 Spark 作业打情感分，输出 Parquet。
- **crawl_bronze**：Glue 爬虫更新数据目录，供 Redshift Spectrum 读取。
- **build_warehouse**：dbt 建 Silver + Gold，并跑 dbt test；测试不过则该步失败、后面不执行，错数据进不了看板。
- **build_snapshot**：查 Gold 出快照。
- **refresh_dashboard**：快照刷进看板。

箭头即依赖，顺序由它决定。各步写入均为覆盖式（幂等），失败重试、按天补跑都安全。

实时这条不用 Airflow，用事件触发：Kinesis 通过 event-source mapping 绑定 Lambda，流里来了记录就自动拉起 Lambda、按微批处理、处理完返回，没有 DAG 也没有定时。它的“调度”就是数据流本身——有数据就触发，并发随分片数扩展。两条链触发方式不同，是因为需求不同：离线要按天成批、讲先后，用 Airflow 按时间和依赖排；实时要来一条尽快处理一条，由流直接驱动，中间再加一层编排器只会添延迟。

## 8. 数据质量

离线链路：本地 jsonl → 上传 S3 → Spark 打分 → dbt 建模 → 看板。

**上传到 S3**

`aws s3 cp --recursive` 是一个个文件传的，整批传没传全它不管——中途断了，或 `--include 'chat_*.jsonl'` 过滤写错漏文件，少传也不报错，结果是数据缺了但看着像成功。

预防：传完对账文件数，再抽查行数，对不上就重传。

```bash
aws s3 ls s3://…/bronze/chat/ --recursive | wc -l   # S3 上的文件数
ls data/chat_*.jsonl | wc -l                         # 本地文件数
```

**Spark 打分（spark_sentiment.py）**

event_info 是嵌套字段、schema 不固定：raid 带 viewerCount，订阅带月数，普通弹幕没有。Spark 读 JSON 要给这列定个统一结构，靠采样几行来推断字段；raid 在 28 万条里就几条，采样很容易一行没抽到，推出来的结构里没有 viewerCount，写出时这 key 就被丢，下游 raid 人数整列是空。

预防：不让它推断，整列转成字符串原样存，用时再取。

```python
# Spark 端：整列序列化成 JSON 字符串
out.withColumn("event_info", to_json(col("event_info")))
# dbt 端：要哪个 key 现取
json_extract_path_text(event_info, 'viewerCount')
```

**dbt 建模**

事实表存的是外键（room_id→channel_key、user 哈希→user_key），报表靠事实 join 维度查出来。风险是 join 丢行：外键在维度表里找不到对应行时（维度建漏、或两边 key 口径不一致），join 把这条事实丢掉，而且 SQL 不报错——少几行是合法结果，静默偏小，看结果发现不了。

预防：在 schema.yml 给键加 dbt 测试，`dbt test` 把它编译成 SQL 查违例，有就判失败。

```yaml
- name: channel_key            # 维度主键：唯一、非空
  tests: [unique, not_null]
- name: channel_key            # 事实外键：必须能在维度里找到
  tests:
    - relationships: { to: ref('dim_channel'), field: channel_key }
```

这些测试在 DAG 里排在出快照、刷看板之前，没过就停，错数据进不了看板。

**实时（dashboard_lambda.py）**

Lambda 按时间窗口把弹幕聚成每频道指标，写进 DynamoDB（key = 频道+窗口），大屏每 2 秒轮询读。问题在读这侧：大屏一开始直接 `table.scan()` 拉全表，但 DynamoDB 的 scan 单次只返回 1MB 一页，boto3 的 `table.scan()` 又不自动翻页；表一大，最近写的数据不在第一页，过滤完就成 0。（CLI 的 `aws dynamodb scan` 反而会自动翻页查得到，容易让人以为数据还在。）

预防两手：

- **读**：`while` 循环带 `ExclusiveStartKey` 翻页，读全所有页再过滤。
- **写**：每条带 `expire_at = now + 1800`，对这个属性开 TTL，半小时前的数据自动删，表一直保持很小。

**幂等**：各步写入都是覆盖式（dbt 整表 drop+create、Spark 覆盖当天分区），同一天重跑不会重复，DAG 的失败重试和按天补跑才安全。
